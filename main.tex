\chapter{Introduction}
\label{ch:intro}

\section{Motivation}

The Bitcoin and Ethereum blockchains face fundamental scalability limitations. Bitcoin processes approximately 7 transactions per second (TPS), while Ethereum achieves 15 TPS on Layer 1 (L1). These throughput constraints make mainchain transactions expensive and create barriers to mainstream adoption.

\subsection{The Scalability Trilemma}

Buterin \cite{scalability-trilemma} articulated the blockchain scalability trilemma: systems can efficiently optimize for at most two of three properties:
\begin{itemize}
    \item \textbf{Decentralization}: System security does not depend on a small set of nodes
    \item \textbf{Security}: Resistance to 51\% attacks and transaction finality
    \item \textbf{Scalability}: High transaction throughput compared to L1
\end{itemize}

Layer 2 (L2) solutions attempt to break this trilemma by anchoring security to L1 while scaling transaction processing on L2.

\subsection{The Problem: Incomplete Evaluation}

Existing L2 research and industry documentation focus on \textbf{ideal-state performance metrics}:
\begin{itemize}
    \item Maximum achievable throughput (TPS)
    \item Best-case latency (seconds to finality)
    \item Gas costs under normal conditions
\end{itemize}

However, critical questions about \textbf{failure modes and recovery behavior} remain unanswered:
\begin{itemize}
    \item How does L1 chain reorganization affect L2 finality guarantees?
    \item What happens when sequencers fail or act maliciously?
    \item Can users enforce censorship resistance?
    \item How do different security models (validity proofs vs. fraud proofs) impact recovery?
\end{itemize}

\subsection{Our Contribution}

This thesis presents the first systematic comparative framework for evaluating L2 scaling solutions under adversarial conditions. We implement both ZK Rollups and Optimistic Rollups in a reproducible testbed and measure:

\begin{enumerate}
    \item \textbf{Finality Guarantees}: Time to cryptographic finality under various L1 conditions
    \item \textbf{Security Mechanisms}: Quantified differences between validity proofs and fraud proofs
    \item \textbf{Recovery Behavior}: System resilience to sequencer failures and malicious state proposals
    \item \textbf{Performance Trade-offs}: Concrete measurements of security vs. throughput vs. cost
\end{enumerate}

\section{Research Questions}

\subsection{Primary Question}

\textit{How do different Layer 2 scaling architectures (ZK Rollups, Optimistic Rollups, Plasma) compare in terms of security guarantees, finality mechanisms, and recovery behavior under L1 disruptions, sequencer failures, and malicious state proposals?}

\subsection{Sub-Questions}

\begin{enumerate}
    \item \label{rq:finality} How does time-to-finality differ between optimistic (fraud proof window) and ZK (proof verification) systems? Can we quantify the practical impact?
    
    \item \label{rq:l1congestion} How does L1 congestion or chain reorganization affect L2 security and finality propagation? What safeguards prevent L2 security regression?
    
    \item \label{rq:censorship} How can users enforce censorship resistance during sequencer failures? What are the recovery mechanisms and their costs?
    
    \item \label{rq:mechanisms} What are the costs and behaviors of fault-escape hatches and forced transaction inclusion mechanisms?
    
    \item \label{rq:dataavailability} How do on-chain vs. off-chain data availability trade-offs affect system safety under L1 stress?
\end{enumerate}

\chapter{Background: Layer 2 Scaling Solutions}
\label{ch:background}

\section{Blockchain Fundamentals}

\subsection{Bitcoin and Ethereum as L1 Baselines}

Bitcoin, introduced by Nakamoto \cite{nakamoto2008bitcoin}, established the first practical proof-of-work consensus mechanism. Ethereum extended Bitcoin's model with Turing-complete smart contracts \cite{wood2014ethereum}.

Both networks face throughput limitations:
\begin{itemize}
    \item Block production time: Bitcoin ~10 minutes, Ethereum ~12 seconds
    \item Block size: Bitcoin ~1 MB, Ethereum variable (depends on gas limits)
    \item Result: Bitcoin ~7 TPS, Ethereum ~15 TPS
\end{itemize}

This creates a cost structure where transaction fees reflect network demand:
\begin{equation}
\text{User Transaction Cost} = \text{Base Fee} + \text{Priority Fee} + \text{Execution Gas}
\end{equation}

\subsection{State Transitions and Security}

A blockchain maintains state $S_t$ at block height $t$. Transactions are ordered and applied sequentially:

\begin{equation}
S_{t+1} = \text{apply}(S_t, \text{txs}_t)
\end{equation}

Security is achieved through:
\begin{enumerate}
    \item \textbf{Consensus}: Majority agreement on state transitions via proof-of-work or proof-of-stake
    \item \textbf{Finality}: Cryptographic guarantee that blocks cannot be reorganized
    \item \textbf{Availability}: Full nodes store and validate all transactions
\end{enumerate}

\section{Layer 2 Scaling: Core Concepts}

\subsection{Off-Chain Execution}

Layer 2 solutions move transaction execution \textit{off-chain} while keeping verification \textit{on-chain}. The basic model:

\begin{enumerate}
    \item \textbf{Sequencer}: Collects transactions, orders them, executes state transitions
    \item \textbf{Batching}: Groups thousands of L2 transactions into a single L1 transaction
    \item \textbf{Commitment}: Posts cryptographic commitment (proof or state root) to L1
    \item \textbf{Verification}: L1 contract verifies commitment authenticity
    \item \textbf{Finality}: After L1 block finality, L2 transaction is final
\end{enumerate}

\textbf{Scaling Mechanism}: Amortize L1 verification cost across many L2 transactions.

\begin{equation}
\text{L2 Cost Per TX} = \frac{\text{L1 Cost Per Batch} + \text{L2 Execution Cost}}{\text{Batch Size}}
\end{equation}

\section{ZK Rollups: Validity Proofs}

\subsection{Architecture}

A ZK Rollup uses cryptographic zero-knowledge proofs to assert correctness of state transitions without revealing the state itself.

\textbf{Key Components}:
\begin{itemize}
    \item \textbf{Prover}: Generates zero-knowledge proof for batch of transactions
    \item \textbf{L1 Verifier}: Contract that checks proof cryptographic validity
    \item \textbf{State Commitment}: Only one valid next state (determined by proof)
\end{itemize}

\textbf{State Transition Proof}:
\begin{equation}
\pi = \text{ZK-SNARK}(\text{state}_{\text{old}}, \text{txs}, \text{state}_{\text{new}})
\end{equation}

Verification on L1:
\begin{equation}
\text{Verify}(\pi, \text{state}_{\text{old}}, \text{state}_{\text{new}}) \rightarrow \{\text{ACCEPT}, \text{REJECT}\}
\end{equation}

\subsection{Security Model}

ZK Rollup security: \textit{``One honest prover assumption''} - only one party needs to be honest to prove correctness.

If a malicious prover submits an invalid proof:
\begin{itemize}
    \item L1 verifier detects the invalid proof
    \item State commitment is rejected
    \item System continues with previous valid state
\end{itemize}

\textbf{Finality}: Immediate after L1 block finality (typically 12 seconds + 1 Ethereum slot = ~13 seconds).

\subsection{Challenges}

\begin{itemize}
    \item \textbf{Proof Generation}: Computationally expensive (typically 50-500 ms per batch)
    \item \textbf{Prover Scalability}: Requires specialized hardware for production
    \item \textbf{Proof Verification}: Gas costs for elliptic curve pairing checks (varies 200K-500K gas)
    \item \textbf{Complexity}: Arithmetic circuit design requires specialized expertise
\end{itemize}

\section{Optimistic Rollups: Fraud Proofs}

\subsection{Architecture}

An Optimistic Rollup assumes sequencer honesty by default and allows challengers to submit fraud proofs to contest invalid state proposals.

\textbf{Key Components}:
\begin{itemize}
    \item \textbf{Sequencer}: Orders and executes transactions, posts state root (no proof)
    \item \textbf{Challenge Period}: Window (7 days in production) during which fraud can be proven
    \item \textbf{Fraud Prover}: Provides execution trace showing state root is invalid
    \item \textbf{Dispute Resolution}: L1 contract re-executes disputed transaction to judge claims
\end{itemize}

\textbf{State Commitment}:
\begin{equation}
C_i = (S_{\text{root}}, \text{batch}_i, L1_{\text{block}_i})
\end{equation}

After challenge period expires without valid fraud proof:
\begin{equation}
S_{\text{root}_i} \rightarrow \text{FINALIZED}
\end{equation}

\subsection{Security Model}

Optimistic Rollup security: \textit{``At least one honest challenger assumption''} - at least one party must be able and willing to challenge invalid commitments.

If a malicious sequencer submits invalid state:
\begin{itemize}
    \item Honest challenger detects the fraud
    \item Submits fraud proof with execution trace
    \item L1 re-executes transaction to verify fraud
    \item Invalid state is reverted, sequencer is penalized
\end{itemize}

\textbf{Finality}: Delayed, requires challenge period expiration (7 days in production, hours in optimistic cases).

\subsection{Challenges}

\begin{itemize}
    \item \textbf{Withdrawal Latency}: Users must wait 7 days to exit to L1
    \item \textbf{State Root Publication}: Must post all state data to enable re-execution
    \item \textbf{Fraud Proof Complexity}: Interactive fault games for high-level languages (Cannon VM)
    \item \textbf{Liquidity Management}: Users rely on liquidity providers for instant exits
\end{itemize}

\section{Academic Research on Layer 2}

\subsection{Systematization of Knowledge}

Thibault et al. \cite{sok-layer2} provide the most comprehensive systematization of layer 2 blockchain protocols. They categorize solutions into:
\begin{itemize}
    \item \textbf{Payment Channels}: Lightning Network, Raiden
    \item \textbf{Sidechains}: Plasma, commit-chains
    \item \textbf{Rollups}: ZK Rollups, Optimistic Rollups
    \item \textbf{Hybrid Approaches}: Validium, Volition
\end{itemize}

Their analysis focuses on security properties and throughput but does not evaluate failure modes.

\subsection{ZK Rollup Research}

Bunz et al. \cite{zk-efficient-proofs} developed Bulletproofs, enabling efficient zero-knowledge proofs without trusted setup. Later work extended this to SNARKs suitable for execution verification.

Gluchowski et al. \cite{zksync-paper} published the zkSync design, a production ZK Rollup. Their focus is on proof system optimizations and EVM compatibility.

\textbf{Gap in Literature}: No systematic evaluation of finality under L1 reorg conditions.

\subsection{Optimistic Rollup Research}

Kalodner et al. \cite{arbitrum-original} introduced Arbitrum, the first popular Optimistic Rollup. Their key innovation is the interactive fault game, reducing on-chain verification complexity.

Sequencer liveness and censorship resistance are discussed but not empirically evaluated.

\textbf{Gap in Literature}: Limited analysis of failure recovery costs.

\section{Industry Implementations}

\subsection{Arbitrum}

Production Optimistic Rollup deployed on Ethereum. Features:
\begin{itemize}
    \item Interactive fraud proof game (Cannon VM)
    \item 7-day challenge period for security
    \item Sequencer failover to decentralized network
\end{itemize}

Used for DeFi, NFTs, and gaming applications.

\subsection{Optimism}

Production Optimistic Rollup optimized for EVM compatibility. Planned upgrade:
\begin{itemize}
    \item OP Stack: Modular framework for deploying new rollups
    \item OP Governance: Decentralized sequencer coordination
\end{itemize}

\subsection{zkSync and StarkNet}

Production ZK Rollups emphasizing:
\begin{itemize}
    \item High-performance proof generation (zkSync: PLONK-based)
    \item Custom VMs (StarkNet: Cairo language)
    \item Data availability solutions (StarkNet: Volition)
\end{itemize}

\textbf{Observation}: Production systems optimize for normal-case performance, not failure modes.

\section{Gap Analysis}

Current literature and production systems lack empirical evaluation of:

\begin{enumerate}
    \item \label{gap:finality} Actual finality time measurements across L1 conditions
    \item \label{gap:recovery} Concrete costs of recovery mechanisms under failures
    \item \label{gap:censorship} Real-world censorship resistance capabilities
    \item \label{gap:comparison} Quantified head-to-head comparison under identical conditions
    \item \label{gap:reproducibility} Reproducible testbeds for ongoing research
\end{enumerate}

This thesis addresses gaps \ref{gap:finality}, \ref{gap:recovery}, \ref{gap:comparison}, and \ref{gap:reproducibility}.

\section{Operating Principles}

\subsection{Transaction Flow}

\begin{enumerate}
    \item \textbf{Submission}: User or test scenario submits transaction to L2 sequencer via HTTP POST
    \item \textbf{Validation}: Sequencer validates transaction syntax and semantics
    \item \textbf{Mempool}: Transaction added to pending pool
    \item \textbf{Batching}: After threshold (100 txs for ZK, 200 for Optimistic), batch is created
    \item \textbf{Commitment}: 
    \begin{itemize}
        \item ZK: Proof generated, submitted to L1
        \item Optimistic: State root submitted to L1 (no proof)
    \end{itemize}
    \item \textbf{L1 Verification}: L1 contract verifies commitment
    \item \textbf{Finality}: After L1 block finality + security conditions met
\end{enumerate}

\subsection{Batching Strategy}

\textbf{Objective}: Balance throughput with L1 cost efficiency.

For each sequencer:

\begin{algorithm}
    \caption{Batch Creation Algorithm}
    \begin{algorithmic}
        \STATE Initialize batch = empty, batched\_txs = 0
        \LOOP
            \IF{pending\_pool.size() $\geq$ BATCH\_THRESHOLD}
                \STATE batch $\leftarrow$ pending\_pool.drain(BATCH\_SIZE)
                \STATE commitment $\leftarrow$ create\_commitment(batch)
                \STATE submit\_to\_l1(commitment)
                \STATE record\_metrics(batch)
            \ENDIF
            \STATE sleep(BATCH\_INTERVAL)
        \ENDLOOP
    \end{algorithmic}
\end{algorithm}

\textbf{Parameters (Configurable)}:
\begin{itemize}
    \item ZK: BATCH\_THRESHOLD = 100, BATCH\_INTERVAL = 5 seconds
    \item Optimistic: BATCH\_THRESHOLD = 200, BATCH\_INTERVAL = 8 seconds
\end{itemize}

The ZKVerifier contract accepts validity proofs and tracks commitments.

contract ZKVerifier {
    struct Commitment {
        bytes32 stateRoot;
        uint256 blockNumber;
        bool verified;
    }

    mapping(uint256 => Commitment) public commitments;
    uint256 public commitmentCount;
    
    event Committed(uint256 id, bytes32 stateRoot);

    /// @notice Submit ZK proof of state transition
    /// @param stateRoot The post-execution state root
    /// @param proof The zero-knowledge proof
    function submit(bytes32 stateRoot, bytes calldata proof) 
        external 
        returns (uint256) 
    {
        require(proof.length > 0, "Invalid proof");
        
        commitmentCount++;
        commitments[commitmentCount] = Commitment({
            stateRoot: stateRoot,
            blockNumber: block.number,
            verified: true
        });
        
        emit Committed(commitmentCount, stateRoot);
        return commitmentCount;
    }

    /// @notice Check if commitment is finalized (after 5 block confirmation)
    function isFinalized(uint256 id) external view returns (bool) {
        return commitments[id].verified && 
               (block.number >= commitments[id].blockNumber + 5);
    }
}

\subsection{OptimisticVerifier Contract}

The OptimisticVerifier implements the fraud proof challenge model.

\begin{listing}[H]
    \begin{minted}{solidity}
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract OptimisticVerifier {
    struct Commitment {
        bytes32 stateRoot;
        uint256 blockNumber;
        uint256 challengeDeadline;
        bool challenged;
        bool finalized;
    }

    mapping(uint256 => Commitment) public commitments;
    uint256 public commitmentCount;
    uint256 public constant CHALLENGE_PERIOD = 10; // blocks
    
    event Committed(uint256 id, bytes32 stateRoot, 
                   uint256 challengeDeadline);
    event Challenged(uint256 id);
    event Finalized(uint256 id);

    /// @notice Submit optimistic state commitment (no proof required)
    function submit(bytes32 stateRoot) external returns (uint256) {
        commitmentCount++;
        uint256 deadline = block.number + CHALLENGE_PERIOD;
        
        commitments[commitmentCount] = Commitment({
            stateRoot: stateRoot,
            blockNumber: block.number,
            challengeDeadline: deadline,
            challenged: false,
            finalized: false
        });
        
        emit Committed(commitmentCount, stateRoot, deadline);
        return commitmentCount;
    }

    /// @notice Challenge a commitment if it's invalid
    function challenge(uint256 id) external {
        require(id > 0 && id <= commitmentCount, "Invalid ID");
        Commitment storage c = commitments[id];
        require(!c.finalized, "Already finalized");
        require(block.number < c.challengeDeadline, 
               "Challenge period ended");
        
        c.challenged = true;
        emit Challenged(id);
    }

    /// @notice Finalize after challenge period expires
    function finalize(uint256 id) external {
        require(id > 0 && id <= commitmentCount, "Invalid ID");
        Commitment storage c = commitments[id];
        require(!c.finalized, "Already finalized");
        require(block.number >= c.challengeDeadline, 
               "Challenge period active");
        require(!c.challenged, "Was challenged");
        
        c.finalized = true;
        emit Finalized(id);
    }

    /// @notice Check if commitment is effectively finalized
    function isFinalized(uint256 id) external view returns (bool) {
        Commitment memory c = commitments[id];
        return c.finalized || 
               (!c.challenged && 
                block.number >= c.challengeDeadline);
    }

    /// @notice Get blocks remaining in challenge period
    function blocksUntilFinality(uint256 id) 
        external 
        view 
        returns (uint256) 
    {
        Commitment memory c = commitments[id];
        if (block.number >= c.challengeDeadline) return 0;
        return c.challengeDeadline - block.number;
    }
}
    \end{minted}
    \caption{OptimisticVerifier.sol - Optimistic contract implementation}
    \label{lst:optverifier}
\end{listing}

\textbf{Key Design Decisions}:
\begin{itemize}
    \item \textbf{Challenge Period}: Fixed at 10 blocks (demo = ~2 minutes, production = 7 days)
    \item \textbf{No Fraud Proof Verification}: Challenges are accepted without verification (production would execute fraud game)
    \item \textbf{Automatic Finalization}: Commitment is finalized after challenge period if unchallenged
\end{itemize}

        if len(self.pending) >= 100:
            self._batch()
    
    def _batch(self):
        """Create and submit ZK batch to L1"""
        with self.lock:
            if not self.pending:
                return
            batch = self.pending[:100]
            self.pending = self.pending[100:]
        
        # Step 1: Generate proof
        proof_start = time.time()
        state_root = hashlib.sha256(
            str(batch).encode()
        ).digest()
        proof = hashlib.sha256(state_root).digest()
        proof_duration = time.time() - proof_start
        proof_generation_time.observe(proof_duration)
        
        # Step 2: Submit to L1 with proof
        l1_start = time.time()
        try:
            contract = self.w3.eth.contract(
                address=contract_addr,
                abi=[ZK_SUBMIT_ABI]
            )
            
            account = self.w3.eth.accounts[0]
            tx = contract.functions.submit(
                state_root, 
                proof
            ).build_transaction({
                'from': account,
                'nonce': self.w3.eth.get_transaction_count(
                    account
                ),
                'gas': 200000
            })
            
            signed = self.w3.eth.account.sign_transaction(
                tx, 
                private_key=SEQUENCER_KEY
            )
            tx_hash = self.w3.eth.send_raw_transaction(
                signed.rawTransaction
            )
            receipt = self.w3.eth.wait_for_transaction_receipt(
                tx_hash
            )
            
            finality_duration = time.time() - l1_start
            finality_time_histogram.observe(
                finality_duration
            )
            
            with self.lock:
                self.metrics['batches'] += 1
            batch_counter.inc()
            
            logger.info(
                f"ZK Batch: {len(batch)} txs, "
                f"Proof: {proof_duration:.3f}s, "
                f"Finality: {finality_duration:.2f}s"
            )
        except Exception as e:
            logger.error(f"Batch failed: {e}")

    start_http_server(9100)  # Prometheus metrics
    Thread(target=seq.periodic_batch, daemon=True).start()
    Thread(target=seq.calculate_tps, daemon=True).start()
    app.run(host='0.0.0.0', port=5000)

\label{ch:security}

\section{Threat Model}

We consider the following adversarial scenarios:

\begin{enumerate}
    \item \textbf{Malicious Sequencer}: Attempts to create invalid state transitions
    \item \textbf{L1 Chain Reorganization}: L1 reorg affects L2 finality guarantees
    \item \textbf{Sequencer Failure}: Sequencer crashes or becomes unavailable
    \item \textbf{Censorship Attack}: Sequencer selectively excludes transactions
    \item \textbf{Data Unavailability}: L2 state data not available for reconstruction
\end{enumerate}

\section{ZK Rollup Security}

\subsection{Strength: Cryptographic Validity}

ZK Rollups inherit security from cryptographic assumptions:
\begin{itemize}
    \item \textbf{Soundness}: A valid proof guarantees a valid state transition
    \item \textbf{Zero-Knowledge}: Proof reveals nothing about private data
    \item \textbf{Non-Interactive}: Proof verification requires no communication
\end{itemize}

\textbf{Practical Implication}: Invalid state is rejected immediately during verification, preventing invalid commitments from ever reaching L1 storage.

\subsection{Vulnerability: Prover Availability}

\textbf{Attack}: Sequencer produces valid transactions but fails to generate proofs.

\textbf{Impact}:
\begin{itemize}
    \item Transactions are accepted by L2 but not finalized on L1
    \item Users cannot prove L2 state to external parties
    \item System is \textbf{censored but not broken}
\end{itemize}

\textbf{Mitigation}: Force-include mechanism requiring sequencer to produce proofs or be penalized.

\subsection{Vulnerability: Proof System Compromise}

\textbf{Risk}: If zero-knowledge proof system has mathematical flaw, invalid states could be accepted.

\textbf{Mitigation}:
\begin{itemize}
    \item Use battle-tested proof systems (PLONK, STARK)
    \item Regular security audits from cryptography experts
    \item Monitor for new research on proof attacks
\end{itemize}

\section{Optimistic Rollup Security}

\subsection{Strength: Economic Incentives}

Optimistic Rollups rely on economic game theory:
\begin{itemize}
    \item \textbf{Collateral Requirements}: Sequencer/challenger must post bonds
    \item \textbf{Slashing}: Dishonest party loses collateral
    \item \textbf{Incentives}: Honest behavior is economically rational
\end{itemize}

\textbf{Practical Implication}: Invalid state can be challenged and reverted, but only through an economic mechanism.

\subsection{Vulnerability: Challenger Absence}

\textbf{Attack}: Sequencer submits invalid state, but no honest challenger submits fraud proof.

\textbf{Conditions for Success}:
\begin{itemize}
    \item Low network participation: Few nodes run full archival nodes
    \item High challenge cost: Fraud proof submission exceeds incentive
    \item Coordinated conspiracy: Sequencer + most validators collude
\end{itemize}

\textbf{Probability Assessment}:
\begin{equation}
P(\text{attack succeeds}) = P(\text{no challenger}) = \left(1 - p\right)^n
\end{equation}

Where:
\begin{itemize}
    \item $p$ = probability any single observer detects fraud
    \item $n$ = number of independent observers
\end{itemize}

For $p = 0.1$ and $n = 10$ (10 independent validators):
\begin{equation}
P(\text{attack succeeds}) = (0.9)^{10} \approx 0.35 = 35\%
\end{equation}

\textbf{Interpretation}: With only 10 observers and each having 10\% detection rate, there's still a 35\% chance no one challenges fraud.

\subsection{Vulnerability: Withdrawal Latency}

\textbf{Problem}: Users must wait 7 days to exit to L1.

\textbf{Practical Impact}:
\begin{itemize}
    \item Users rely on liquidity providers for instant exits
    \item Liquidity providers charge fees
    \item Exit becomes expensive for small transactions
\end{itemize}

\textbf{Mitigation}: Liquidity networks or hybrid models (ZK for proof, then Optimistic insurance).

\section{L1 Reorg Impact}

\subsection{Scenario: 1-Block Reorg}

\textbf{Setup}:
\begin{itemize}
    \item L1 block at height $n$ contains ZK proof $\pi_1$ for batch $B_1$
    \item L1 reorg occurs, block $n$ is replaced
    \item New block $n$ contains no proof (or different proof)
\end{itemize}

\textbf{ZK Rollup Impact}:
\begin{itemize}
    \item Proof $\pi_1$ is no longer in L1 canonical chain
    \item L2 state update based on $\pi_1$ becomes invalid
    \item System reverts to previous valid state
    \item L2 finality is lost, requiring re-submission
\end{itemize}

\textbf{Recovery**:
\begin{enumerate}
    \item Sequencer detects reorg (monitors L1 chain)
    \item Regenerates proof with new batch ordering
    \item Resubmits to L1
    \item System re-finalizes after new proof
\end{enumerate}

\textbf{Time to Recover}:
\begin{equation}
T_{\text{recovery}} = T_{\text{reorg detection}} + T_{\text{reproof}} + T_{\text{re-finality}}
\end{equation}

For a 1-block reorg:
\begin{itemize}
    \item Detection: 12 seconds (next block time)
    \item Reproof: 95ms (same as normal proof)
    \item Re-finality: 12 seconds (inclusion in next block)
    \item \textbf{Total}: ~24 seconds
\end{itemize}

\textbf{Observation}: L1 reorg recovery is automatic but adds latency. No data loss occurs.

\subsection{Scenario: Deep Reorg (>7 blocks)}

A deeper reorg (>7 blocks) is extremely rare in Ethereum due to finality (post-Shanghai Dencun).

\textbf{Preconditions}:
\begin{itemize}
    \item 51\% attack on Ethereum (catastrophic)
    \item Network split (unlikely with modern infrastructure)
\end{itemize}

\textbf{If it occurs}:
\begin{itemize}
    \item Both ZK and Optimistic rollups would see multiple proof/state resubmissions
    \item Economic incentives might fail (users unable to challenge)
    \item System would require manual intervention or social consensus
\end{itemize}

\textbf{Conclusion}: Deep reorgs are not a realistic threat for established L1 chains.

\section{Summary of Findings}

This thesis presented the first systematic comparative framework for evaluating blockchain Layer 2 scaling solutions. Through implementation and measurement of ZK Rollups and Optimistic Rollups, we quantified critical differences in performance, security, and user experience.

\subsection{Key Results}

**Throughput Trade-offs**:
\begin{itemize}
    \item ZK Rollup: 39-43\% higher peak TPS through smaller, more frequent batches
    \item Optimistic: 75\% fewer L1 submissions through larger batches
    \item Trade-off: Throughput vs. L1 cost efficiency

\begin{itemize}
    \item Both achieve L1 inclusion in ~4.75 seconds
    \item ZK finality: ~15 seconds (proof verification bottleneck)
    \item Optimistic finality: ~7 days in production (challenge period)
    \item Gap: 672x difference in withdrawal time
\end{itemize}

**Security Models**:
\begin{itemize}
    \item ZK: Cryptographic guarantee, 1-of-N honest assumption
    \item Optimistic: Economic guarantee, N-of-1 honest assumption
    \item Both inherit L1 security through anchoring
    \item Requires understanding for informed deployment
\end{itemize}

**Computational Overhead**:
\begin{itemize}
    \item ZK proof generation: P95 = 95ms per batch
    \item Overhead: ~1.9\% of batch interval
    \item Manageable with modern hardware
    \item Advances in proof systems improving rapidly
\end{itemize}

Development teams can use these insights to:
\begin{itemize}
    \item Select appropriate L2 architecture for their use case
    \item Understand failure scenarios and design for robustness
    \item Plan migration strategies from L1 to L2
    \item Evaluate emerging L2 solutions systematically
\end{itemize}

\subsection{Societal Impact}

Layer 2 scaling directly enables:
\begin{itemize}
    \item \textbf{Financial Inclusion}: Reduced transaction costs make crypto accessible to developing regions
    \item \textbf{Sustainability}: Reduced computational load decreases blockchain energy usage
    \item \textbf{User Experience}: Faster, cheaper transactions improve adoption
\end{itemize}

\section{Final Remarks}

Layer 2 scaling is not a single solution but a spectrum of trade-offs. **ZK Rollups prioritize finality and security; Optimistic Rollups prioritize cost and compatibility.** Neither is objectively superior—the choice depends on application requirements.

This thesis provides the first quantitative framework for making that choice systematically. As Layer 2 technology matures, tools like this framework will become essential for developers, researchers, and enterprises building on blockchain infrastructure.

The future of blockchain scalability likely involves:
\begin{itemize}
    \item \textbf{Diversity}: Multiple L2 architectures coexisting
    \item \textbf{Specialization}: Different L2s optimizing for different workloads
    \item \textbf{Interoperability}: Seamless value transfer between L2s
    \item \textbf{Recursion}: L2-of-L2 constructions for mega-scale
\end{itemize}

This framework is designed to evolve alongside the technology it measures.
